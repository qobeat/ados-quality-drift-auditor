## Context

The project needs a complete next-iteration instruction that preserves v0.1 safeguards and prevents the control-to-execution drift observed in prior ADOS Glossary builds.

## Decision

The following prompt is the controlling starting point for production-grade v1.0. Any change to its mandatory gates requires an owner-approved ADR.

# PRODUCTION-GRADE V1.0 BUILD PROMPT

## Role

Act as the architect, implementation lead, verification engineer, and release preparer for the **ADOS Quality Drift Auditor**. Build the production-grade version without weakening the v0.1 contracts to obtain an easier PASS.

## Inputs

1. `ados-quality-drift-auditor-v0.1.zip`
2. The complete evaluation examples and source materials supplied with the v0.1 project
3. Any owner-approved v1.0 amendments
4. No hidden internet authority unless the owner explicitly adds it as a source

## Target

Produce:

- `ados-quality-drift-auditor-v1.0.zip`
- `ados-quality-drift-auditor-v1.0-build-summary.json`
- independent audit report;
- mutation-test report;
- deterministic replay report;
- migration report from v0.1 to v1.0.

## GOAL

Deliver a production-grade, deterministic, schema-governed project-evolution quality auditor that can ingest one or many versions of a project, evaluate build controls, building-agent work, and build results under a selected archetype profile, trace quality drift and its consequences, select the best baseline, and produce independently auditable JSON and human-readable reports.

## OBJECTIVES

1. Preserve the v0.1 claim-ledger, evidence-separation, archetype, metric, problem-bank, hard-gate, and baseline-selection contracts.
2. Implement a safe command-line workflow for source-set initialization, package inventory, report validation, comparison, and report rendering.
3. Support all three input classes for one or many versions with explicit missing-evidence behavior.
4. Make archetype selection evidence-based, explainable, and testable without reducing it to keyword matching.
5. Implement strict JSON Schema Draft 2020-12 validation for every active JSON instance and every generated report.
6. Provide deterministic claim-weight, metric-weight, score, evidence-coverage, hard-gate, ranking, and verdict calculations.
7. Implement cross-version and cross-stage drift tracing for all governed drift dimensions.
8. Expand and calibrate the problem bank using positive, negative, ambiguous, and non-applicable fixtures.
9. Add safe archive handling, path normalization, resource limits, and contamination checks.
10. Prove validator defect sensitivity with isolated mutations and exact expected finding codes.
11. Add an independent release auditor that does not import or trust the primary validator's decision logic.
12. Make the package reproducible from two unrelated clean extractions with byte-identical governed outputs where determinism is required.
13. Provide complete user, maintainer, schema-author, profile-author, and problem-bank-author documentation.
14. Derive the final release verdict only after every mandatory gate completes.

## AUTHORITY AND NON-DRIFT RULES

1. Read `PACKAGE.json`, `MANIFEST.md`, `GLOSSARY.md`, `PDD.md`, all accepted ADRs, the machine skill, banks, schemas, and examples before planning.
2. Treat v0.1 as the contract baseline. Do not silently rename stable IDs.
3. Do not weaken `additionalProperties: false`, required fields, identifier patterns, enums, or cross-record validation to make existing examples pass.
4. Any intentional breaking schema change requires:
   - a new ADR;
   - a migration rule;
   - before/after fixtures;
   - explicit owner approval;
   - compatibility impact in the change log.
5. Preserve the distinction between observation, derived fact, inference, judgment, and unconfirmed claim.
6. Preserve the exact phrase `I cannot confirm this.` for decisive unsupported propositions.
7. Do not infer release PASS from a sample, fixture, test count, file count, schema-binding count, or builder-owned validator PASS.
8. Do not select the latest version as baseline without applying the baseline policy.
9. Register all implementation files before execution. Unplanned material files require a PDD amendment.
10. Keep source artifacts, extracted work trees, generated reports, and release artifacts in separate declared locations.

## REQUIRED ARCHITECTURE

### A. CLI and library

Provide an installable standard-library-first Python package or an explicitly justified dependency set with locked versions.

Required commands:

```text
ados-qda init <source-set.json> --out <report.json>
ados-qda validate-package <package-root-or-zip>
ados-qda validate-report <report.json>
ados-qda inventory <artifact> --out <inventory.json>
ados-qda compare <source-set.json> --out-dir <out/>
ados-qda render <evaluation-report.json> --format markdown
ados-qda self-test
ados-qda audit-release <package-root-or-zip>
```

Each command must have:

- documented exit codes;
- structured JSON stdout option;
- deterministic output ordering;
- no writes outside an explicit output or temporary directory;
- clear behavior for partial evidence.

### B. Safe source ingestion

Implement:

- ZIP path traversal prevention;
- symlink policy;
- decompressed-size and file-count limits;
- binary/text classification;
- SHA-256 inventory;
- media-type recording;
- no execution of untrusted source content during inventory;
- explicit opt-in for project-specific test execution;
- cleanup of temporary workspaces.

### C. Archetype system

Retain all v0.1 profiles. For each profile add:

- competency questions;
- minimum evidence matrix;
- archetype-specific positive and negative fixtures;
- calibration examples;
- profile version and change history;
- tie-breaking tests;
- profile validation and reference resolution.

Archetype selection must output ranked candidates, confidence, evidence, exclusions, and final rationale. A declared archetype is an input, not automatic truth.

### D. Claim and evidence engine

Implement deterministic support for:

- stable source and version identifiers;
- atomic claim creation and editing;
- dependency validation;
- claim-weight calculation and normalization to exactly 100.0 per source;
- claim-score and weighted-score calculation;
- evidence ledger validation;
- stale and broken evidence detection;
- independence classification;
- source/claim/evidence referential integrity.

Semantic claim extraction may remain reasoning-agent assisted, but the package must validate and calculate the resulting ledger deterministically.

### E. Metric and verdict engine

Implement:

- profile-specific stage weights;
- fixed common axes;
- null handling without midpoint substitution;
- evidence-coverage calculation;
- hard-gate evaluation;
- stage score and whole-document score;
- comparison deltas;
- ranking;
- baseline recommendation checks;
- final verdict derivation.

The engine must reject:

- weights not totaling 100;
- weighted scores inconsistent with source scores;
- PASS below policy thresholds;
- PASS with a failed hard gate;
- PASS with insufficient evidence;
- unknown metric, problem, archetype, claim, evidence, source, or version references.

### F. Drift engine

Support comparisons:

- baseline result to build package;
- build package to agent work;
- agent work to build result;
- build package to build result;
- result to later result.

For each drift dimension record observation, direction, severity, evidence, consequence, risk, and corrective action.

Implement explicit checks for:

- goal and scope drift;
- contract weakening;
- semantic synchronization;
- identity and inventory drift;
- evidence staleness;
- validator sensitivity regression;
- replay-order and hidden-state drift;
- dependency changes;
- package contamination.

### G. Problem bank

Retain all v0.1 problems and add:

- stable taxonomy;
- aliases and related patterns;
- applicability conditions;
- false-positive exclusions;
- detection confidence;
- default severity with project-specific override rules;
- remediation verification;
- version history.

Real examples must remain traceable to supplied evidence and must not be generalized beyond their observed boundary.

### H. Reports

Generate:

1. source normalization;
2. evidence ledger;
3. atomic claim ledger;
4. archetype-selection record;
5. stage evaluations;
6. common-axis evaluations;
7. semantic claim alignment;
8. drift findings;
9. problem matches;
10. baseline recommendation;
11. final evaluation JSON;
12. human-readable Markdown report.

JSON remains the score and verdict authority. Rendering must not recalculate or change results.

### I. Validator and independent auditor

The primary package validator must check:

- required files and folders;
- exact manifest inventory;
- hash policy;
- schema meta-validation;
- complete active JSON governance;
- declared instance bindings;
- reference integrity;
- profile weight totals;
- metric weight totals;
- stable ID uniqueness;
- report arithmetic;
- verdict-policy conformance;
- absolute-path and cache contamination;
- production prompt presence.

The independent auditor must separately reimplement at least:

- exact inventory and hashes;
- schema and binding validation;
- weight totals;
- score arithmetic;
- hard-gate/verdict consistency;
- reference integrity;
- mutation result verification.

It must not import the primary validator module.

## REQUIRED MUTATION SUITE

Run every mutation in an isolated fresh copy and require the exact finding code.

At minimum:

1. add an unregistered file;
2. corrupt a manifest hash;
3. weaken a critical schema;
4. change a profile stage weight total to 99;
5. reference an unknown metric;
6. reference an unknown problem;
7. delete an evidence record used by PASS;
8. set PASS below score threshold;
9. set PASS with failed hard gate;
10. fabricate an artifact hash;
11. use a missing version ID in a comparison;
12. introduce an absolute path;
13. add `__pycache__` or `.pyc`;
14. create a ZIP traversal entry;
15. generate a claim-weight total other than 100;
16. make weighted scores inconsistent with source scores;
17. change the selected baseline without a rationale;
18. generalize a fixture PASS to package PASS;
19. break the production prompt marker;
20. introduce an undeclared active JSON file.

Preserve mutation input, change, expected finding, observed finding, stdout, stderr, and exit code.

## REQUIRED REAL-EXAMPLE REGRESSION TESTS

Create non-copyrighted minimal fixtures representing the observed ADOS Glossary failure classes:

- v1.4-style structure expansion with weak contracts;
- work plan with no concrete targets;
- semantic registry/export mismatch;
- validator accepting critical mutations;
- v1.5-style complete binding with permissive schemas;
- assertion-only evidence;
- proof created before final package;
- builder-owned all-PASS verdict;
- calculator fixture PASS combined with unrelated package defects;
- older strong baseline versus newer structurally richer but weaker version.

The expected result must identify the correct problem IDs, hard gates, and baseline outcome.

## EVIDENCE REQUIREMENTS

Every mandatory verification record must include:

- check ID;
- procedure or command;
- expected result;
- observed result;
- target artifact and hash;
- UTC timestamp;
- verifier and independence class;
- stdout/stderr or structured output path;
- result;
- supported claim or gate IDs.

The target path alone is not evidence.

## RELEASE SEQUENCE

```text
load authority
→ define PDD and registered file set
→ create one-file work items
→ implement
→ file-level verification
→ schema and reference validation
→ unit and integration tests
→ real-example regression
→ isolated mutation suite
→ two clean-extraction deterministic runs
→ primary package validation
→ independent audit
→ report rendering comparison
→ final Reflection review
→ derive verdict
→ create release ZIP and external SHA-256
```

Do not create proof artifacts before the final candidate exists when the proof claims final-artifact replay.

## ACCEPTANCE GATES

All are mandatory for PASS:

1. All JSON schemas meta-validate under Draft 2020-12.
2. Every active JSON instance is bound to a schema or has a narrowly approved exemption.
3. Every declared binding validates.
4. All stable IDs are unique and references resolve.
5. Every metric and profile weight frame totals exactly 100.
6. Calculation tests cover rounding, nulls, evidence coverage, hard gates, and verdict precedence.
7. Primary validator rejects every required mutation with the exact expected code.
8. Independent auditor rejects the same release-breaking mutations through separate logic.
9. Real-example regression identifies the intended v1.4/v1.5 problem patterns.
10. The simple calculator example is classified as bounded evidence and never yields package PASS by itself.
11. Safe ZIP tests reject traversal and resource-limit violations.
12. Two clean extractions produce identical deterministic governed outputs.
13. Fresh extraction contains no unregistered files, absolute paths, cache files, or stale identities.
14. All CLI commands have tested success and failure exit codes.
15. Reports validate and human rendering preserves exact scores and verdict.
16. Documentation and machine authority agree.
17. No unresolved critical or high-severity release blocker remains.
18. Independent audit returns PASS.
19. Final evidence coverage is at least 0.90 for production release claims.
20. Final verdict is derived after all gates.

## NON-PASS BEHAVIOR

If a mandatory gate fails:

- do not weaken the gate or schema;
- preserve the failure evidence;
- create focused corrective work;
- rerun only after correction;
- return FAIL or CONDITIONAL_PASS according to policy;
- list unresolved blockers and exact next actions.

## FINAL RESPONSE

Report:

- artifact links and SHA-256;
- source baseline and migration summary;
- file and folder counts;
- schema and binding counts;
- archetype/profile counts;
- problem-pattern counts;
- test counts by class;
- mutation results;
- deterministic replay results;
- independent audit result;
- evidence coverage;
- unresolved blockers;
- final score and verdict.

Do not state “production-grade” unless every acceptance gate passes.


---

## Consequences

- v1.0 has an explicit contract rather than a vague “improve this package” request.
- Release validation and independent audit are distinct.
- Real v1.4/v1.5 problem examples become regression fixtures.
- The build cannot claim production status from internal PASS alone.
