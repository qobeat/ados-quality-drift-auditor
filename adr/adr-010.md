## Context

A future v1.0 build needs a complete instruction that preserves the glossary-first and non-governance boundaries.

## Decision

The following prompt is the controlling starting point for v1.0. Material changes to its Goal or non-drift constraints require owner approval and a superseding ADR.

# FULL PROMPT — Produce production-grade ADOS Quality Drift Auditor v1.0

## Role

Act as architect, terminology engineer, evaluator-method designer, schema designer, and release reviewer.

## Inputs

1. A clean extraction of `ados-quality-drift-auditor-v0.2`.
2. Owner-approved amendments.
3. Representative multi-version Project examples covering all six Archetypes and primary Domains.
4. No hidden external authority.

## Goal

Produce a production-grade `ados-quality-drift-auditor-v1.0.zip` that performs glossary-centered, Archetype-aware, Domain-aware Project-evolution quality and drift evaluation while remaining non-invasive toward the evaluated agent and Project.

## Objectives

1. Preserve `GLOSSARY.md` as semantic authority.
2. Review every glossary definition for genus, differentia, boundary, cardinality, relations, auditor use, and drift checks.
3. Validate all glossary references from profiles, metrics, problems, skill, and reports.
4. Calibrate Archetype and Domain selection using varied positive, negative, ambiguous, and mixed examples.
5. Define Coordinate Dimensions and Metrics from Project Goals and Objectives, with explicit scales, polarity, target conditions, evidence expectations, and anti-substitution guardrails.
6. Implement deterministic JSON Schema validation for all active JSON surfaces.
7. Add a safe evaluator host implementation only if it does not impose execution governance on evaluated Projects.
8. Support incomplete source sets and return explicit uncertainty.
9. Produce direct JSON evaluation reports and faithful human-readable renderings.
10. Detect and explain stage drift and cross-version drift.
11. Select the strongest Baseline Version rather than automatically selecting the latest.
12. Produce focused corrective prompts preserving User Request, Intent, Goal, and Objectives.
13. Expand Problem Patterns with calibrated exclusions and remediation verification.
14. Provide optional semantic exports only if they remain synchronized with GLOSSARY.md.
15. Demonstrate quality on held-out Projects without using one known fixture as universal proof.

## Non-drift constraints

- Do not introduce mandatory claim, evidence, execution, or work ledgers.
- Do not require evaluated Projects to use PDD, Atomic Work, Directives, or ADOS packaging.
- Do not convert Metrics into release gates.
- Do not redefine Glossary terms in other files.
- Do not reintroduce app-wide terminology where Project is the general concept.
- Do not treat file counts, schema counts, test counts, or self-reported PASS as sufficient quality evidence.
- Do not use hidden chain-of-thought as required evidence.

## Required v1.0 surfaces

- PACKAGE.json
- README.md
- MANIFEST.md
- GLOSSARY.md
- PDD.md
- CHANGE-LOG.md
- accepted ADRs
- adapted doctrine under doc/
- machine skill
- Archetype and Domain banks
- Coordinate and Metric bank
- report policy
- Problem Pattern bank
- complete JSON schemas and schema index
- report templates and diverse examples
- optional host implementation with clearly separated source/artifact workspaces
- validation and independent review outputs external to the semantic authority

## Required evaluator behavior

1. Normalize Project, versions, and available input types.
2. Retrieve relevant glossary terms.
3. Identify or reconstruct User Request, Intent, Goal, and Objectives only from supplied evidence.
4. Select one Archetype and one Domain with confidence and rationale.
5. Build the active Semantic Meaning Space and Goal-Attractor Geometry.
6. Evaluate Build Package, Building Agent Work, and Build Result when available.
7. Compare stages and versions coordinate by coordinate.
8. Match Problem Patterns only from observed signals.
9. Separate confirmed consequence from inferred risk.
10. Recommend a Baseline Version and explain rejected alternatives.
11. Produce a focused Corrective Prompt.
12. State `I cannot confirm this.` for decisive unsupported propositions.

## Required evaluation examples

At minimum include:

- App / Business Project;
- Research / Science Project;
- Runtime / Software-Engineering Project;
- Profile / Governance Project;
- Prompt / Education or Business Project;
- Other custom Project;
- missing Build Package;
- missing agent work;
- older strong baseline versus newer weaker version;
- semantic improvement with structural simplification;
- structural growth with semantic degradation;
- false-PASS source report;
- proof fixture generalized beyond its direct behavior;
- app-specific Requirement incorrectly applied to another Archetype.

## Quality checks

- All JSON schemas meta-validate.
- All active JSON instances validate.
- Every glossary term reference resolves.
- Archetype and Domain IDs are unique.
- Metric and Coordinate references resolve.
- Problem Pattern term and metric references resolve.
- Report examples validate.
- Score arithmetic and confidence semantics are consistent.
- Missing evidence remains explicit.
- Reports do not contain release PASS/FAIL as the project-evolution conclusion.
- Human renderings preserve the JSON assessment.
- Real examples demonstrate improvement, mixed movement, degradation, unchanged, and insufficient evidence.
- Independent review confirms the package evaluates rather than governs agent execution.

## Acceptance

Do not call v1.0 production-grade unless:

- semantic review finds no unresolved critical concept contradiction;
- all required examples are represented;
- schema and reference checks pass;
- Archetype and Domain selection is calibrated;
- Goal-Attractor Coordinates are interpretable and non-duplicative;
- Problem Patterns have observable signals and exclusions;
- report outputs are faithful and actionable;
- no mandatory execution-governance process has been introduced.

## Deliverables

- `ados-quality-drift-auditor-v1.0.zip`
- build summary
- concept migration report from v0.2
- glossary consistency report
- schema/reference validation report
- Archetype/Domain calibration report
- Metric calibration report
- diverse example evaluation set
- independent architecture review


---

## Consequences

- v1.0 has an explicit semantic and architecture contract.
- Production claims require diverse examples and independent review.
- Host implementation remains subordinate to the Glossary and evaluator boundary.
