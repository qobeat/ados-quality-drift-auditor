# ADR-007 — Keep doctrine in doc/ and decisions in adr/

## Metadata

| Attribute | Value |
|---|---|
| DOCUMENT_TYPE | architecture_decision_record |
| DOCUMENT_STATUS | accepted |
| UPDATED_AT | 2026-06-19 |
| DOCUMENT_ROLE | document_role_decision |
| GOVERNANCE_AREA | architecture |
| CONSUMERS | owner; architect; evaluator; agent; reviewer |
| WHEN_TO_USE | Use when interpreting the architectural choice recorded by this ADR. |
| HOW_TO_USE | Read Context, Decision, Alternatives, Consequences, and Scope. Doctrine details remain in doc/ and terms remain in GLOSSARY.md. |
| DOCUMENT_CONTENT | Decision separating explanatory doctrine from accepted architecture decisions. |


## Context

Source ADOS packages contain substantial methodology documents. Converting every doctrine document into an ADR would blur explanation and binding choice.

## Decision

- `doc/` explains concepts, models, methodology, and adaptations.
- `adr/` records accepted architecture choices for this Project.
- `GLOSSARY.md` owns term meaning.
- ADRs may cite docs but do not duplicate them wholesale.

## Alternatives

- Move all docs into ADR records.
- Keep decisions only in README or PDD.

## Consequences

- Agents can distinguish explanation from decision authority.
- ADRs remain concise.
- Source doctrine can evolve without silently changing accepted architecture.
